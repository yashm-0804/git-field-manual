The whole of Git, laid out so you can find it fast.

Git tracks the history of your files. GitHub is the website that stores a copy online so you can back it up and collaborate. This manual walks the path a real project takes — set up, save work, branch, sync, and recover when things break — with the most-used commands flagged and the dangerous ones clearly marked.

★ = you'll use daily
⚠ = can delete work

# The mental model

01

Before any command makes sense, you need the picture. Your work moves through four places. Almost every command you'll ever run just moves a file from one box to the next. Learn this diagram and 80% of Git stops being mysterious.

The one sentence to rememberadd picks which changes go in the next save, commit saves them into your history with a message, and push uploads that history to GitHub.

# Setup & identity

02

Do this once per computer. Git stamps every save with your name and email, so it needs to know who you are. You also need a way to prove your identity to GitHub when uploading.

git --version
Confirms Git is installed. No output means you need to install it from git-scm.com (Windows/Mac) or via your package manager.
git config --global user.name "Your Name"
Sets the name attached to your commits. --global means "for every project on this machine."
git config --global user.email "you@example.com"
Use the same email as your GitHub account so your commits link to your profile.
git config --list
Shows all your current settings, so you can check the name and email actually saved.
git config --global init.defaultBranch main
Makes new repositories start on a branch called main (the modern default GitHub uses).
Authentication — read this onceGitHub no longer accepts your account password from the command line. When you push over HTTPS it asks for a Personal Access Token instead of a password — create one under GitHub → Settings → Developer settings → Tokens. The easiest path for beginners is to install GitHub CLI (gh auth login) or GitHub Desktop, which handles login for you.

# Start a project

03

There are exactly two ways to begin: init turns a folder you already have into a Git project, and clone downloads an existing project from GitHub.

git init
★ key
Run inside a folder to start tracking it. Creates a hidden .git folder that holds all history. Do this when the project lives only on your computer so far.
git clone https://github.com/user/repo.git
★ key
Downloads a full copy of a GitHub project — code and its entire history — into a new folder. This is how you grab someone else's (or your own) existing repo.
terminal
$ git clone https://github.com/octocat/Hello-World.git
Cloning into 'Hello-World'...
remote: Enumerating objects: 7, done.
$ cd Hello-World   # now you're inside the project

# The daily cycle ★

04

This is the loop you'll repeat hundreds of times: change files → check → stage → commit → push. If you only memorise one section, make it this one.

git status
★ most used
Your compass. Shows what's changed, what's staged, and what branch you're on. Run it constantly — there's no harm, and it always tells you the next step.
git add filename.txt
★ key
Stages one file — marks it to be included in the next commit. Think of it as putting an item in the box before sealing it.
git add .
★ most used
Stages every changed file in the folder at once. The everyday shortcut. (The . means "here and below.")
git commit -m "Describe what you changed"
★ most used
Permanently saves the staged changes into your history with a short message. The message is a note to future-you about why the change happened.
Good messages are short and present-tense: "Add login button", not "changed stuff".
git commit -am "message"
Stages and commits all already-tracked files in one step. Skips git add — but only works for files Git already knows about (not brand-new files).
git push
★ most used
Uploads your committed history to GitHub. Until you push, your work lives only on your machine.
git push -u origin main
first push
The first push of a new branch. -u links your local branch to GitHub so that afterwards a plain git push just works.
git pull
★ most used
Downloads and merges the latest changes from GitHub into your branch. Run it before you start working, especially on a shared project, so you're up to date.
The full loop in practice
a normal work session
$ git pull                       # 1. get latest
# ... you edit files ...
$ git status                     # 2. see what changed
$ git add .                      # 3. stage it all
$ git commit -m "Add nav bar"    # 4. save with a message
$ git push                       # 5. upload to GitHub

# Branches & merging

05

A branch is a parallel copy of your project where you can experiment without touching the main version. You build a feature on its own branch, then merge it back into main when it's ready.

git branch
Lists all branches; a * marks the one you're currently on.
git switch -c feature-name
★ key
Creates a new branch and moves you onto it. The modern, clearer replacement for git checkout -b.
git switch main
★ key
Moves you to an existing branch. Your files instantly change to match that branch.
git checkout branch-name
The older command that also switches branches. You'll see it everywhere online; switch does the same job more safely.
git merge feature-name
★ key
Brings the named branch's changes into the branch you're on now. So to merge a feature into main: switch to main first, then merge.
Order matters: git switch main → then → git merge feature-name.
git branch -d feature-name
⚠ deletes
Deletes a branch after it's merged. Use a capital -D to force-delete an unmerged branch.
Capital -D throws away unmerged work permanently. Be sure it's merged first.

# Remotes & sync

06

A remote is just a nickname for a copy of your repo that lives elsewhere — almost always on GitHub. The default nickname is origin. These commands connect your local project to GitHub and keep the two in sync.

git remote add origin https://github.com/you/repo.git
★ key
Links a local project (made with git init) to an empty GitHub repo. Run this once, then push. origin becomes the name you push to.
git remote -v
Shows which GitHub URL your project is connected to. Handy when a push goes to the wrong place.
git fetch
safe
Downloads the latest changes from GitHub but doesn't touch your files yet. Lets you look before you merge. (A pull is really a fetch + merge.)
git push -u origin branch-name
Publishes a new local branch to GitHub for the first time and remembers the link, so later you just type git push.

# Undoing mistakes

07

Everyone breaks things. The trick is knowing which undo to reach for. They range from totally safe to permanently destructive — the ⚠ badges below tell you which is which.

git restore filename.txt
⚠ loses edits
Throws away your uncommitted changes to a file, resetting it to the last commit. Useful when an experiment went wrong — but the edits are gone for good.
git restore --staged filename.txt
safe
Un-stages a file you added by mistake. Your edits stay; it just removes the file from the next commit. The opposite of git add.
git commit --amend -m "Corrected message"
Rewrites your most recent commit — to fix a typo in the message or add a forgotten file. Only do this if you haven't pushed yet.
Already pushed? Amending changes history and will conflict with GitHub. Prefer a new commit instead.
git revert commit-hash
safe to share
The safe undo for work already on GitHub. It creates a brand-new commit that cancels out an old one, leaving history intact. Best choice on shared projects.
git reset --soft HEAD~1
⚠ rewrites
Undoes the last commit but keeps your changes staged, as if you never committed. Great for "oops, I committed too early." HEAD~1 means "one commit back."
git reset --hard HEAD~1
⚠⚠ destroys work
Deletes the last commit and every change in it — files and all. There's usually no getting it back.
The most dangerous everyday command. Never run reset --hard unless you're certain you want those changes gone forever.

# Inspecting history

08

Git remembers everything. These read-only commands let you look back — what changed, when, and who did it. None of them alter anything, so explore freely.

git log
★ key
Lists past commits — newest first — with author, date, and message. Press q to quit the viewer.
git log --oneline --graph
A compact, one-line-per-commit view with a visual branch graph. Far easier to scan than the full log.
git diff
★ key
Shows the exact lines you've changed but not yet staged. Green is added, red is removed. Run it before committing to review your work.
git show commit-hash
Displays everything about one specific commit — its message and all the lines it changed.
git blame filename.txt
Shows who last changed each line of a file and in which commit. Less about blame, more about "why is this line here?"

# Stashing

09

You're mid-change and suddenly need to switch branches, but you're not ready to commit. Stash tucks your unfinished work onto a shelf so your folder goes clean — then you bring it back later.

git stash
safe
Saves all uncommitted changes onto a hidden shelf and resets your folder to clean. Nothing is lost.
git stash pop
safe
Brings your shelved changes back and removes them from the shelf. The usual way to resume.
git stash list
Shows everything currently on the shelf, in case you've stashed more than once.

# Forks & pull requests

10

This is the GitHub-the-website part. A fork is your personal copy of someone else's repo. A pull request (PR) is how you propose your changes to be added back into the original — the heart of collaboration and open source.

Fork vs Clone — the common confusionFork happens on the GitHub website (a button) and makes a copy under your account. Clone happens on your computer and downloads a repo locally. You usually fork first, then clone your fork.
# on github.com: open a Pull Request
After pushing your branch, GitHub shows a "Compare & pull request" button. A PR is a request to merge your changes, where others can review and comment first.
git remote add upstream https://github.com/original/repo.git
Adds a second remote pointing at the original project (your fork's source). Lets you pull in their latest updates with git pull upstream main.
Try GitHub CLIOnce installed, gh repo clone, gh pr create, and gh pr list let you do GitHub actions without leaving the terminal — and gh auth login solves the password/token headache in one step.

# .gitignore

11

Some files should never go to GitHub — passwords, huge dependency folders, machine-specific junk. A .gitignore file lists patterns Git should pretend don't exist.

# create a file named .gitignore in your project root
Put one pattern per line. Anything matching is left out of every commit. Create it early — once a file is committed, ignoring it later is harder.
.gitignore
# dependencies — never upload these
node_modules/
venv/

# secrets & environment
.env
*.key

# OS & editor noise
.DS_Store
.vscode/
Why it mattersCommitting a .env file with API keys to a public repo is one of the most common beginner mistakes — bots scan GitHub for leaked keys within minutes. Add it to .gitignore before your first commit.

# Common recipes

12

Real situations you'll hit, with the exact steps. Search this section by the problem you're facing.

# Recipe: resolve a merge conflict
When two people change the same lines, Git pauses and marks the clash with <<<<<<< markers. Open the file, pick the correct version, delete the markers, then:
after editing the file
$ git add the-conflicted-file.txt
$ git commit                  # finishes the merge
# Recipe: I committed to the wrong branch
Undo the commit (keeping changes), switch to the right branch, and re-commit there.
fix wrong-branch commit
$ git reset --soft HEAD~1      # undo commit, keep changes
$ git switch correct-branch
$ git commit -m "your message"
# Recipe: my push was rejected
It means GitHub has commits you don't. Someone pushed before you. Pull their work first, then push again.
push rejected fix
$ git pull        # merge their changes in
$ git push        # now it works
# Recipe: update my fork from the original
Pull the original project's latest commits into your fork (assumes you added an upstream remote).
sync a fork
$ git fetch upstream
$ git switch main
$ git merge upstream/main
$ git push            # update your fork on GitHub

# Glossary

14

Every confusing word, in plain English.

repository
A project tracked by Git — your folder plus its full history. Often shortened to "repo."
commit
A saved snapshot of your project at a point in time, with a message describing it.
branch
A parallel line of work. main is the default trunk; you create others to experiment safely.
HEAD
A pointer to "where you are right now" — usually the latest commit on your current branch.
staging area
The waiting room. Files you've add-ed sit here until the next commit.
remote
A copy of your repo hosted elsewhere, usually on GitHub.
origin
The default nickname for your main remote (the GitHub repo you cloned or pushed to).
clone
Downloading a full copy of a remote repo to your computer.
fork
Your own copy of someone else's GitHub repo, made on the website.
pull request
A proposal on GitHub to merge your changes into another repo, open to review.
merge
Combining the changes from one branch into another.
conflict
When two changes touch the same lines and Git needs you to choose which wins.
hash
The unique ID of a commit, like a3f9c2e. You only need the first 7 characters.
upstream
The original repo your fork was made from.
